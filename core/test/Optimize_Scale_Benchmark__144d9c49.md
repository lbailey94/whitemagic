### Key Optimizations
- **Batch SQLite inserts** via `executemany` with WAL + MEMORY temp store — bypasses tool API middleware entirely
- **Embedding optional** (`embed=False` default) — skips FastEmbed model loading for pure FTS5 testing
- **Direct FTS5 search** — bypasses cross-encoder and semantic rerankers for sub-5ms latency at 100K scale
- **Combined search+recall pass** — single query loop measures both latency and recall simultaneously
- **Bug fixed**: `break` statement was prematurely terminating the row-building loop after 1 memory per batch (only 10 memories inserted instead of 10K)

### Observations
- **Search latency stays sub-25ms** even at 100K — FTS5 BM25 scales well
- **Recall degrades with scale** — R@1 drops from 53% (10K) to 41.5% (100K) as more similar memories compete
- **FTS5 alone is insufficient** — semantic reranking (FastEmbed) and cross-encoder would improve recall significantly but add latency
- **Insert throughput slows at ~78K** — WAL checkpoint causes a brief throughput drop (15K → 8K ops/sec)
- **Total runtime**: 10K in <1s, 50K in ~4s, 100K in ~13s — well within the "not hours" requirement
