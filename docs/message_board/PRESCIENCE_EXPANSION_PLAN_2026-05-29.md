## 2. Claim Discovery Strategy

### 2.1 Primary Archives (highest yield)

| Archive | Size | Yield Estimate | Method |
|---------|------|---------------|--------|
| CODEX/OpenAI archives | 220+ conversations | 20–40 claims | Parse for dated predictions, technology forecasts, policy assertions |

| Twitter archives | 10K+ tweets | 5–10 claims | Short-form predictions with public resolution dates |
| WhiteMagic git history | 2+ years | 10–15 claims | Architecture decisions that predated industry convergence |

**Total addressable claims: 70–130.** This would push the Brier score toward statistical significance (n > 30 is the threshold for calibration reliability).

### 2.2 Discovery Pipeline

```
Phase 1: Ingest → Phase 2: Extract → Phase 3: Score → Phase 4: Validate
```

**Phase 1: Ingest**
- Load all Markdown conversations into a searchable index (Fragment already exists but is stale)
- Normalize dates: every claim must have a `source_date` (when the prediction was first uttered)
- Deduplicate: same claim in multiple archives = one canonical source

**Phase 2: Extract (automated + human)**
- Use LLM pass: "Find all dated predictions in this conversation. For each, return: claim_text, source_date, predicted_outcome, confidence_level, category."
- Human review: filter for claims with independently verifiable validation events (public announcements, product launches, policy changes)
- Reject: vague claims ("AI will be big"), claims without dates, claims without verifiable resolution criteria

**Phase 3: Score**
- Assign confidence using both stated probability and behavioral confidence (tone analysis)
- Compute expected points: `lead_weeks_estimate * confidence_adjustment`
- Prioritize: high-confidence + long-lead + verifiable resolution

**Phase 4: Validate**
- For each pending claim, set a validation search query
- Weekly automated scan: Google Alerts / Exa search for validation events
- Human confirmation: verify the source matches the prediction criteria
- Update `TemporalForecastDB` with validation_date and validation_ref

### 3.3 Automated Validation Scanner

**The missing piece:** Currently, validation is manual. We need:

```python
# core/whitemagic/forecasting/validation_scanner.py
class ValidationScanner:
    """Weekly scan for validation events matching pending claims."""
    
    def scan(self, claim: Prediction) -> ValidationEvent | None:
        """Search the web for validation of a pending claim."""
        query = self._build_query(claim)
        results = exa_search(query, start_date=claim.source_date)
        return self._score_match(claim, results)
```

**Integration:**
- Run weekly via cron or Celery beat
- Use Exa MCP tool for semantic web search
- Post matches to Sangha `#prescience` channel for human review

## 4. Improving the Prescience System

### 4.1 Calibration Improvements

Current calibration uses 5 bins. With more claims, we should:
- Increase to 10 bins for finer-grained calibration
- Add **reliability diagram** (confidence vs. empirical frequency)
- Add **ROC curve** for binary claims (validated vs. falsified)
- Track **lead-time decay**: does accuracy decrease with longer lead times?

### 4.2 Category-Specific Scoring

Different categories have different base rates:
- AI governance: fast-moving, high validation rate
- Geopolitics: noisy, lower base rate
- Science: slow, high confidence when validated

**Solution:** Compute category-specific Brier scores and BSS. This prevents high-volume categories from dominating the aggregate score.

### 4.3 Behavioral Confidence Integration

Current system stores both `confidence` (stated) and `behavioral_confidence` (post-hoc tone analysis). We should:
- Use behavioral confidence as the primary score for Brier calculation
- Track the gap between stated and behavioral as a "meta-calibration" metric
- If the gap is systematic (e.g., always under-reporting), apply a correction factor

### 4.4 Monte Carlo Projection

For pending claims, simulate possible outcomes:

```python
def project_points(pending_claims, n_simulations=10000):
    """Monte Carlo simulation of expected point gain."""
    total_points = []
    for _ in range(n_simulations):
        sim = sum(
            c.points if random() < c.confidence else 0
            for c in pending_claims
        )
        total_points.append(sim)
    return {
        "mean": mean(total_points),
        "p10": percentile(total_points, 10),
        "p90": percentile(total_points, 90),
    }
```

This gives a probability distribution over future points, not just a point estimate.

## 6. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Brier score regresses with more claims | Medium | Expect regression; focus on category-specific scores |
| False positives in automated validation | High | Human review gate; auto-validate only at > 0.9 confidence |
| Overfitting to past claims | Medium | Reserve 20% of claims as a holdout test set |
| Source date ambiguity | Medium | Use earliest verifiable mention; document uncertainty |
| Validation bias (only validating wins) | High | Explicitly track falsified claims; maintain falsification rate |

## 7. Connection to WhiteMagic Systems

The prescience system is not standalone. It connects to:

- **Sangha:** Validation events are logged as community insights
- **Karma Ledger:** Each claim resolution is an auditable event
- **PRAT:** `forecast.evaluate` is a dispatchable tool (future)

**The vision:** WhiteMagic doesn't just track predictions. It tracks *how* predictions are made, scores the methodology, and publishes the audit trail. This is the "Foresight Engine" as a first-class citizen.
