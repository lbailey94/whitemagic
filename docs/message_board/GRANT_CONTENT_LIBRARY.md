## Block O: Construct Validity Argument (Schmidt Sciences / Scientific Funders)

```
## Construct Validity

**Construct**: [What you are measuring — one sentence, precise]
**Why it matters**: [Connection to real-world risk or decision]
**How we capture it**: [Measurement procedure, step by step]
**Why this measure is valid**:
- Face validity: [Experts agree this is the right thing to measure]
- Content validity: [Covers all relevant cases / tool types / scenarios]
- Concurrent validity: [Correlates with an established measure or human judgment]

**Example (Karma Ledger)**:
- Construct: Side-effect fidelity — the degree to which an agent's declared tool-use effects match its empirically observed effects.
- Why it matters: If an agent declares "read file X" but writes to file Y, oversight systems fail. Current benchmarks test task success (LoCoMo) or memory accuracy (LongMemEval-S) but not declaration fidelity.
- How we capture it: For each tool call, compare declared JSON schema against actual system-state diff. Score = 1 - (normalized edit distance between declared and actual state).
- Why valid:
  - Face validity: Safety researchers agree declaration-actual mismatch is a critical gap.
  - Content validity: Covers all MCP tool types (file, browser, API, database, shell).
  - Concurrent validity: Pilot correlation with human judgment of "suspicious" tool calls should be measured before submission.
```
